# **AI 执行上下文授权模型（白话版）**

## **一、这个协议想解决什么问题**

这个模型解决一个核心问题：

**AI 在运行时，什么时候允许做“外部动作”？**

例如：

- 调用 API
- 访问数据库
- 写文件
- 发消息
- 控制设备

这些行为都会产生 **外部副作用**。

如果没有约束，AI 可以：

- 任意调用能力
- 在推理过程中不断扩大权限
- 产生不可追溯的行为

这个协议的目标是保证：

1. **AI 的能力上限在任务开始时就确定**
2. **推理过程中不能扩大能力**
3. **任何外部行为都必须先申请能力**
4. **能力必须被验证**
5. **每个外部行为都必须能追溯到一次授权**

简单说：

> AI 可以思考很多，但只有经过授权的能力才能影响外界状态。

## **二、核心概念**

这个模型里有几个最重要的概念。

### **1 Execution Context（执行上下文）**

**Execution Context 可以理解为：**

> 一次完整的 AI 任务执行。

例如：

- 一次用户提问
- 一次 agent 任务
- 一次自动流程

这个任务从 **开始到结束** 就是一个 Context。

特点：

- 有开始
- 有结束
- 内部是一段连续推理

例如：

```
用户请求 -> 创建 Context
AI 推理
AI 调用能力
AI 继续推理
任务完成 -> Context 结束
```

### **2 推理步骤（Step）**

AI 的执行被拆成 **很多步骤**：

例如：

```
Step1 推理
Step2 推理
Step3 请求能力
Step4 推理
Step5 输出
```

协议把整个执行看成：

```
一条时间序列
```

每一步都有时间顺序。

### **3 Context 的隔离**

协议规定：

**Context 之间不能交叉。**

意思是：

```
Context A
---------
step1
step2
step3

Context B
---------
step4
step5
```

不能出现：

```
A step1
B step1
A step2
B step2
```

也就是说：

**Context 必须是一个连续时间区间。**

这样做是为了避免：

- 推理污染
- 授权混乱
- 责任不清

### **4 Context ID**

每个 Context 都有一个唯一 ID。

这样：

- 所有授权
- 所有能力
- 所有副作用

都可以绑定到 **具体 Context**。

## **三、能力系统**

AI 可以使用很多能力，例如：

- HTTP 请求
- 读文件
- 写文件
- 数据库
- shell

这些能力被拆成 **最小能力单位**：

```
Capability
```

例如：

```
ReadFile
WriteFile
HTTP
SendEmail
```

## **四、Capability Ceiling（能力上限）**

这是协议最核心的机制之一。

当 **Context 创建时**：

系统会计算一个：

```
Capability Ceiling
```

意思是：

> 这个任务 **最多允许使用哪些能力**

例如：

用户请求：

```
读取日志并总结
```

系统可能给出：

```
Ceiling = {
ReadFile
LLM
}
```

如果 AI 想：

```
HTTP 请求
删除文件
```

就会被拒绝。

## **五、能力上限不可改变**

协议规定：

**Ceiling 一旦确定，整个 Context 生命周期不能改变。**

也就是说：

```
任务开始时：
Ceiling = {A,B,C}

之后任何步骤：
Ceiling 都必须是 {A,B,C}
```

不能出现：

```
后来增加权限
```

例如：

```
突然允许 shell
突然允许 root
```

全部禁止。

## **六、能力请求（Request）**

AI 在每一步可以 **请求能力**。

例如：

```
Req(step3) = {ReadFile}
```

意思是：

```
这一步想使用 ReadFile
```

协议规定：

```
请求必须属于 Ceiling
```

即：

```
Req ⊆ Ceiling
```

否则拒绝。

## **七、验证机制**

AI 请求能力之后，需要 **验证（Validation）**。

也就是说：

系统会判断：

```
这个请求是否允许
```

例如：

```
Req = {ReadFile}

验证系统：
允许 / 拒绝
```

如果验证通过：

```
Valid = true
```

否则：

```
Valid = false
```

## **八、外部事件**

任何 **影响外部世界的行为** 都叫：

```
Event
```

例如：

- HTTP 调用
- 写文件
- 发消息
- 调 API

协议要求：

> 每个外部事件必须对应一次执行步骤。

也就是说：

```
某个 step 触发了这个事件
```

不能出现：

```
事件来源不明
```

## **九、能力映射**

每个外部事件必须映射到 **能力集合**。

例如：

```
HTTP request -> {HTTP}
写文件 -> {WriteFile}
```

这样系统可以检查：

```
这个事件需要什么能力
```

## **十、外部事件授权规则**

一个外部事件 **只有满足以下条件才能发生**：

### **条件1**

事件需要的能力必须在 Ceiling 内。

即：

```
Map(event) ⊆ Ceiling
```

### **条件2**

必须存在一个能力请求。

```
Map(event) ⊆ Req(step)
```

### **条件3**

该请求已经通过验证。

```
Valid = true
```

简单说：

**没有请求 → 不允许执行**

**没有验证 → 不允许执行**

**超过能力上限 → 不允许执行**

## **十一、Scope（能力范围）**

Scope 是：

> 当前 Context 允许使用能力的 **子范围**

关系是：

```
Scope ⊆ Ceiling
Req ⊆ Scope
```

也就是说：

```
Ceiling  = 最大权限
Scope    = 当前任务范围
Req      = 当前步骤请求
```

结构是：

```
Req ⊆ Scope ⊆ Ceiling
```

## **十二、信息流控制**

协议还约束了：

**变量之间的依赖关系。**

意思是：

如果变量 A 依赖变量 B：

```
A = f(B)
```

那么系统可以追踪：

```
A 的来源
```

这样可以避免：

- 数据越权
- 信息污染

## **十三、安全闭环**

协议最终形成一个 **安全闭环**。

任何外部行为必须满足：

```
外部行为
   ↓
能力映射
   ↓
能力请求
   ↓
能力验证
   ↓
Context 能力上限
```

如果任何一步不成立：

```
行为被拒绝
```

## **十四、协议保证的四个性质**

这个模型保证四个关键安全属性。

### **1 能力不可扩展**

AI 不能在运行时增加权限。

### **2 副作用可追溯**

任何外部行为都能追溯到：

```
哪个 Context
哪个 Step
哪个 Request
```

### **3 验证先行**

所有能力必须：

```
先验证
再执行
```

### **4 Context 隔离**

不同任务之间：

```
完全隔离
```

不会共享权限。

## **十五、内部推理与外部行为的区别**

AI 内部推理：

```
允许任意进行
```

例如：

- 思考
- 规划
- 推理

这些 **没有外部副作用**。

但如果推理会：

```
改变系统状态
```

仍然需要授权。

## **十六、协议适用范围**

当前 v1.0 有一些限制：

只支持：

```
非嵌套 Context
非并发 Context
```

也就是说：

一次只处理一个任务。

复杂系统如果有嵌套：

需要拆成多个 Context。

## **一句话总结**

这个协议的核心思想是：

**把 AI 的权限控制，从“身份授权”升级为“执行上下文授权”。**

即：

```
权限绑定到任务
而不是绑定到 AI 本身
```

这样可以保证：

- 权限不会在推理中膨胀
- 外部行为完全可追溯
- 每个任务严格受控
